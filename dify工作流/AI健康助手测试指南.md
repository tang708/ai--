# AI健康助手 — 测试指南

---

## 一、系统类型识别

本系统属于 **类型B：RAG型（对话+知识库）**

- 有知识库：运动健康知识库（28项运动）
- 有上下文变量：用户健康数据（API传入）
- 有 Prompt：健康助手人设 + 输出格式规范

**测试重点**（按PDF方法论 A→B→C 优先级）：

| 优先级 | 维度 | 说明 |
|--------|------|------|
| A 必测 | 忠实性 | 运动知识是否基于知识库，不瞎编 |
| A 必测 | 边界意识 | 不知道/没权限的事拒绝回答 |
| A 必测 | 安全性 | 防提示词注入、防信息泄露 |
| B 常测 | 准确性 | 运动参数（时长/心率/频率）是否准确 |
| B 常测 | 完整性 | 健康建议是否覆盖指标→运动→饮食→生活 |
| C 选测 | 一致性 | 同一问题问3次，答案是否自洽 |
| C 选测 | 格式遵循 | 是否禁止Markdown、用编号分段、300字内 |

## 二、测试分层（智能体5层法应用）

| 层级 | 测试内容 | 方法 |
|------|----------|------|
| 输入层 | 错别字、超长、中英混、空白 | 手动测试 |
| 提示词理解层 | 是否按角色+规则+边界回应 | 手动测试+T6用例 |
| 知识检索层 | 检索结果是否匹配问题 | Dify 后台查看检索日志 |
| 输出层 | 7维度检查 | 自动化脚本+人工抽查 |
| 工具调用层 | 本系统无工具调用 | 不适用 |

## 三、测试用例设计（五步法）

### 第一步：问题类型

| 类型 | 类型名 | 测试重点 | 用例数 |
|------|--------|----------|--------|
| T1 | 正常查询 | 忠实+准确 | 10 |
| T2 | 模糊查询 | 忠实+边界 | 5 |
| T3 | 多问题混合 | 完整性 | 3 |
| T4 | 行为请求 | 边界 | 3 |
| T5 | 诱导泄露 | 安全 | 3 |
| T6 | 提示词注入 | 安全 | 3 |
| T7 | 边界兜底 | 边界 | 3 |
| **合计** | | | **30** |

### 第二步：用例设计

见 `AI健康助手测试用例.csv`

### 第三步：判断标准

每条用例有 `must_have`（必须出现）和 `must_not_have`（不能出现）：

```
must_have:   准确答案 → 必须包含这些关键词（|分隔，任一即可）
must_not_have: 错误答案 → 绝对不能包含这些关键词
```

### 第四步：执行

每个用例跑 3 次，记录通过率。通过标准：≥ 2/3。

### 第五步：报告

自动化脚本自动生成分类型通过率、响应时间、失败明细。

## 四、如何测试

### 方式1：Dify 云端手动测试

打开 Dify 应用「AI健康助手」→ 调试与预览 → 逐条输入用例 → 人工判断

**适用场景**：初版快速验证、Prompt 调试

# 修改脚本顶部的 API_KEY 和 API_URL（已填好）

## 六、Bug 归因

根据 PDF 方法论，出现 Bug 时按层级排查：

1. **输出不对但知识库有正确答案** → 知识检索层问题（chunk 切分、检索策略）
2. **输出是知识库原文照搬但格式不对** → Prompt 层问题（输出规范不够明确）
3. **输出瞎编、知识库没有的内容** → Prompt 层问题（忠实性约束不够强）
4. **被注入/越狱成功** → Prompt 层问题（边界声明不够硬）
5. **同一个问题3次答案差异大** → 大模型层（可尝试 temperature=0）
